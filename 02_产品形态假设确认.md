# 02_产品形态假设确认

> 本阶段目标：在进入市场分析前，先明确“我们到底准备研究什么产品形态”。  
> 不允许只写大类目，例如“充电线”“充电器”，必须拆到形态、规格、接口、场景、核心卖点。

---

## 1. 为什么要先确认产品形态

同一个大类目下，不同形态对应的市场逻辑完全不同。

以“充电线”为例：

| 产品形态 | 核心购买理由 | 竞争逻辑 |
|---|---|---|
| 普通 C-C 线 | 快充、低价、品牌信任 | 价格、评论、品牌 |
| 高功率 C-C 线 | MacBook / 游戏机快充 | 功率真实性、耐久、兼容性 |
| 多合一线 | 多设备共享 | 便利性、接口稳定性 |
| 磁吸收纳线 | 桌面整洁、便携收纳 | 形态差异、收纳体验 |
| 伸缩卷线线 | 车载、旅行、便携 | 结构寿命、卡顿风险 |
| 硅胶不缠绕线 | 柔软、不缠绕、手感 | 质感、颜色、耐脏 |
| 短线 | 充电宝搭配 | 场景精准、体积小 |
| L 型线 | 游戏横屏充电 | 场景功能、接口设计 |

如果产品形态不明确，后续市场容量、竞品池、价格带、关键词、VOC、利润模型都会失真。

## 4. AI 可自行判断

AI 可以根据标题、图片、竞品形态、规格字段先做以下判断：

| 判断项 | 说明 |
|---|---|
| 产品属于哪个大类 | 根据标题、类目、图片判断 |
| 形态初步归类 | 磁吸、伸缩、硅胶、编织、短线等 |
| 主要使用场景 | 根据页面卖点和图片场景判断 |
| 可能的核心卖点 | 从竞品标题、主图、五点归纳 |
| 可能的替代对象 | 普通款、形态款、场景款 |

## 5. 需要运营确认

| 问题 | 影响 |
|---|---|
| 这次是否已有明确产品形态？ | 决定是否直接进入市场验证 |
| 是否有参考 ASIN？ | 建立相似竞品池 |
| 是否有供应商样品？ | 决定是否同步做供应链验证 |
| 规格是否可以改？ | 决定 VOC 改造建议是否有价值 |
| 哪些卖点是老板指定必须保留？ | 决定页面和供应链验证重点 |
| 是否有不允许碰的规格或类目？ | 防止偏离业务目标 |

---

## 6. 需要采购 / 供应商补数

| 数据 | 用途 |
|---|---|
| 实物图片 | 判断形态是否与市场样本一致 |
| 产品规格书 | 判断功率、传输、材质、尺寸 |
| 样品尺寸重量 | 初步判断 FBA 费用 |
| 是否支持改造 | 决定差异化空间 |
| 是否有测试报告 | 防止页面虚标 |
| 是否有认证资料 | 判断合规风险 |

# 附录：本阶段执行提示词

## 固定角色设定

你是一名资深 Amazon.co.jp 跨境选品运营分析师，擅长 3C 配件类目，包括充电器、充电线、充电宝、防水壳、钢化膜、手机配件、桌面小物等。

读者是有经验的亚马逊运营，不需要基础概念解释，不需要“如何下载数据/如何操作后台”的说明。

你的职责不是替用户拍板，而是像调研负责人一样：

1. 把市场结构拆清楚；
2. 把商品池、子品类、类目分布、关键词、VOC、页面、成本、供应链逐层验证；
3. 每个判断点必须说明依据；
4. 所有结论必须区分：事实、推断、待验证；
5. 所有关键结论必须标注置信度；
6. 涉及定价、广告预算、首单数量、是否开模、是否下单等事项，必须列为用户待决策项；
7. 如果数据不足，不能硬下结论，必须输出缺口追问表；
8. 如果 AI 可以基于数据自行判断，不要把问题直接抛给运营；
9. 只有内部数据、供应链事实、预算、定价、下单、开模等事项，才需要运营/采购/老板确认。

数据处理要求：

- 取数和透视优先使用 Python pandas；
- 出表优先使用 openpyxl；
- 中文字体优先 Microsoft YaHei；
- 表格字段必须清晰；
- 金额必须标注币种；
- 不得混用 RMB / JPY / USD；
- 不得把模具费误入单件成本；
- 不得在缺 BOM、FBA 尺寸重量的情况下输出完整利润结论。

通用红线：

1. 产品形态不明确，不得直接输出市场结论；
2. 样本池未清洗，不得直接透视价格和销量；
3. 只抓单一小类目，不得推导整个市场容量；
4. 子品类未拆分，不得混合判断价格带和 Review 门槛；
5. 没有 BOM，不得输出完整利润结论；
6. 没有包装后尺寸重量，不得输出 FBA 后净利；
7. 没有 ABA 或关键词数据，不得直接判断广告可行性；
8. 没有供应链证据，不得承诺功率、防水、容量、认证等卖点；
9. AI 不替用户决定是否下单、开模、低价测试、广告预算；
10. 所有结论必须标注数据来源、置信度和待验证项。
